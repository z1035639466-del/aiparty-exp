# aiparty v2.0 收官推进设计

## 目标

按《推进执行手册》把阶段 1 从已冻结的 DM-skill v2.0 推进到可审计的 v2.0 验证批；所有结构性改动都先通过测试、校验器和版本化文档，所有判断性环节保留给指定的人工 SOP 卡点。

## 范围与边界

本轮实现只覆盖手册第 0–6 步中的 Codex 任务和 Opus 归档任务：

1. 核对并入库已成稿的 `docs/specs/DM-skill-v2.0.md`。
2. 回写 C5–C7 与六原语正典；实现 `check.py v2.0` 的硬闸、软闸和件头 warning。
3. 补齐 `inputs/README` 登记册、`run_ds.py` 的 thinking 参数口径和 36 家族三型归属表。
4. 以 Claude Opus 4.8、High 推理强度生成并归档 `04_结算结构与惩罚层.md`；它冻结评级尺 v2.1 和五列打标规程。
5. 依据 04 规程生成试点 `pilot-tags.md`，但不越过用户的试点抽审卡点。
6. 以 `fable_A_v18_r1` 先行生成 goldenset 迁移候选，保留原件，待用户逐键审后才定稿为 `_v20_m1`。
7. 仅当上述前置全部为绿且用户放行时，按预注册编排生成 v2.0 主批、B×20、结构检查和 v0.3 review；重试和 pass@2 必须等 v0.3 人工抽审放行。

不在范围内：修改现有 v1.8 历史 JSON、改变已冻结字段键、替用户完成任一人工抽审、把表达力债登记直接升级为 v3 schema，或提交当前无关的 `outputs/haiku_A_v18_02.json` 缩进改动与缓存目录。

## 交付结构

| 单元 | 职责 | 主要产物 |
| --- | --- | --- |
| 正典与规格 | 固定 v2.0 字段语义和原语边界 | `docs/specs/design-layer-v2.0.md`、`docs/specs/spec-prop-library-v0-final.md`、`docs/specs/DM-skill-v2.0.md` |
| 校验器 | 机械硬闸、非阻断软闸、warning 件头输出 | `check.py`、`tests/test_check.py` |
| 存量登记 | 输入、归属和 DS 运行口径可追溯 | `inputs/README.md`、归属表、`tests/test_run_ds.py` |
| 评级归档 | 冻结净收益尺度和可重复的五列打标格式 | `04_结算结构与惩罚层.md`、`pilot-tags.md` |
| 迁移与验证 | 只结构化既有散文，随后按预注册矩阵验证 | `_v20_m1.json`、`.review3.md`、汇总表 |

## 数据流与模型选择

`DM-skill v2.0` 和 `inputs` 产生候选 JSON；`check.py v2.0` 先施加结构硬闸并把软闸 warning 写入件头；裁判 v0.3 读取 JSON 与 warning，生成只读 review；用户审查指定样本后才允许下一阶段。

| 工作 | 执行者与强度 | 原因 |
| --- | --- | --- |
| 04 规范成稿、尺度措辞冻结 | Claude Opus 4.8 / High | 涉及跨文档一致性与判断性归档 |
| 正典回写、校验器、登记、试点初标、迁移候选 | Codex / 高推理 | 以确定性规格、测试和逐键来源追踪为主 |
| 主批生成 | Sonnet、Haiku 4.5、DeepSeek，各 2 发 | 与 v2.0 预注册矩阵一致 |
| B×20 | DeepSeek | 保持既定 shape 分布终审口径 |

DeepSeek 的 `off`、`on`、`both` 只按 `run_ds.py` 源码选择：分别为非思考作业、仅 thinking/high 作业、两者合并；实际验证批使用哪一条 DS 车道，必须先与既有 v1.8 基线命名一致后锁定并写入汇总口径。

## 错误处理与验证

校验器把“机械可判、误杀率零”的问题作为硬闸错误；把散文/参数数字不一致、死账目、未引用道具作为 warning，不拒件。每个新增校验先以单元测试复现失败，再做最小实现；旧 v1.8 件的三份空转只证明硬闸生效，绝不写入统计。

每个阶段提交前运行针对性测试和完整测试集；涉及 JSON 的阶段同时运行 `check.py`。验证批使用预注册的命名、样本量和停止点，不在用户抽审前计算 pass@2。

## 人工卡点

1. 试点抽审：4 份（2 变动、2 维持，至少 1 条 A 段对照），同意率至少 3/4 才铺开 341 条。
2. 迁移逐键审：用户核对原件散文、候选 `params` 和 `scoring`；只删除发明、补回漏抽。
3. v0.3 抽审：4 份（2 破、2 过，包含指定对账判破和 DS 过件）；放行前禁止重试与 pass@2。
4. 味抽查：每模型 2–3 份，含 1 份 DS 过件。

## 提交策略

在当前 `master` 上以窄提交推进并在每次用户请求的仓库变更提交后立即推送。提交不纳入无关的工作区变更；每个提交只包含一个可验证阶段的规格、实现或归档产物。
