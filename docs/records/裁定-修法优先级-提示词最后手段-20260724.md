# 裁定:修法优先级——提示词是最后手段(2026-07-24)

> 房主原话:"不能出什么问题都去加提示词,加到后面全是问题解决不了,
> 能用其他方法解决(写代码给他调用规范化)就用其他方法解决。"

## 病灶

真机测试进入密集报障期后,修法出现了一个坏习惯:每个"局长行为不对"的病历
都往系统提示词里加一句纠正。单看每句都合理,合起来是死路:

- **按句膨胀**:铁律段落已数千字,每加一句,所有旧句的权重被稀释一分;
- **软执法**:提示词是"请求模型别这样",模型忘了就白写——代码执法不会忘;
- **位置错配**:纠正"读场没骰子就不开骰玩法"的话写在几百行外的常驻段落,
  误导性信号(实物清单)和解药不在同一处,等于没写。

当日实例:骰盅"自我阉割"问题第一版修法是往【道具规则】加三行常驻文本;
第二版改为搬进条件注入的读场块+六字括号注——同样的纠正力,常驻成本归零。

## 修法优先级(上一级走不通才降到下一级)

1. **引擎守卫/钳制层**——行为可检验的规则用代码执法。
   先例:越界调用拒写留痕、束手拍检测(no_closer)、唤醒券补发、三句话硬截断、
   开牌当庭报数(challenge_verdict)、叫价/颗数钳制。零 token,不可遗忘。
2. **工具声明**——只关一个工具的规范写进该工具的 declaration,规则跟着工具走,
   删工具规则自动消失,不污染全局段落。
3. **条件注入**——上下文相关的规则跟着触发它的信号一起出现/消失。
   先例:读场规则"没输入不注水"(有测试锚定)、虚拟道具不受实物清单限制。
4. **常驻提示词**——只留宪法级定位与原则(【定位】【判罚】这一档)。
   加句子前必须先回答"上面三级为什么不行",答不上不许加。

## 边界(诚实条款)

代码管不了模型的**信念**:开什么游戏、怎么理解场面是推理内部的事,没有钩子能
强制"你要考虑 X"。这类问题的代码侧解法是**治信号不治脑子**——把误导它的输入
贴上标签/换个呈现,把解药放在信号旁边(条件注入),而不是指望常驻段落里的
一句话在几百行外遥控纠偏。

## 维护条款

铁律段落再度长胖时,做**蒸馏**(把攒下的病历句合并成更少的原则句),不做追加。
蒸馏是一次专项任务,动手前重读本裁定与产品宪法。
