# golden set v2.0 迁移单（乙脚手架 · 产品侧连带改动之二）

> 授权依据：统一处理令裁决"乙仅许作开发期脚手架（AI 试译→人工审→蒸馏进映射表），永不进生产路径"；
> v2.0 §5 明文"golden set 全量迁移启用乙脚手架，产物只进 golden set 不进生产链"。

## 1. 迁移面清点

- **golden 输入 A–D：不迁**。输入是口味+素材，schema 变的是输出侧。
- **迁移对象 = gold 档案内的样板局**（v1.8 产物）。精确件数以 inputs/README 登记册为准
  （Codex 前置，未落盘不开迁）。已知必迁：`fable_A_v18_r1`（基准件）。
- **fail 档案不迁**——历史卷宗，失效类目录引用的是 v1.8 原件与 v0.2 判词，动了反而毁证据链。
- **v1.9 零迁移量**：从未跑批，无产物。

## 2. 迁移流程（每件三步，禁跳步）

1. **AI 试译**：v1.8 件 → v2.0 件。做两类事且只做两类事：
   ① 把 plain_rule 散文里的载荷参数抽进 rules[].params（按 check.py 规格 §1 分机制表）；
   ② v1.8 件无 settlement 字段——从原件的散文结算描述**转写**出 scoring/win/tiebreak 三键，
   逐条标注来源行。**设计零改动断言：散文一字不动，只结构化；转写不发明原件没有的结算。**
2. **人工逐键审**：对照原件散文核每个 params 键与每条 scoring 条目；发明出来的删，漏抽的补。
3. **定稿**：件名 `_v20_m1`（m=migrated），原件原地留档不动。过 check.py v2.0 + 裁判 v0.3
   双闸后方可登册。

## 3. 基准件特殊条款（fable_A_v18_r1 → fable_A_v20_m1）

- 双审章继承规则：**纸面自玩章直接继承**（玩法未变，迁移是编码变换）；**裁判章须重挣**——
  迁移件过裁判 v0.3 后双章齐，加一行迁移注记，golden 之首地位延续。
- 若 v0.3（判据比 v0.2 严）在迁移件上查出**原设计固有**的新问题：不回改设计、不悄悄修——
  按 SOP 分歧归档格式原样记录，基准件地位由用户终审。测量仪变严暴露旧金的瑕疵是制度在干活，
  不是迁移事故。
- 若查出的问题属**迁移引入**（转写失真）：退回第 2 步重审，不算原件账。

## 4. 蒸馏条款（乙的退场机制）

试译中每个"散文→params"的对应模式，记一行进映射表（给编译站与未来 lint 用）。
迁移完成即乙退场：生产路径上 AI 直接产 v2.0 格式（DM-skill v2.0 正典 prompt），不存在翻译环节。
