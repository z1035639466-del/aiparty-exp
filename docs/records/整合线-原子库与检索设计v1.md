# 整合线 · 原子库与检索设计 v1(2026-07-18)

> 回答:采集来的几万条数据怎么最大化利用、怎么变成 AI 能用的东西、RAG 怎么建。
> 定位:序列钉死中「整合 → 写 → 建库调用」三步的施工图。消费方三个:
> 狂野模式 `draw_atom`(运行时)、Mode A 生成佐料(反同质化)、正典/机制库扩编。
> 性质:设计案,schema 已用真实数据 50 条抽样验证(见 §6);待房主照准后批量施工。

## 0. 总原则:资产是结构化原子库,RAG 只是它上面的一层索引

「建 RAG」不是把几万条原文扔进向量库。向量检索答不了运行时真正的问题——
「4 个人、没道具、野度≤6、这局没用过、别跟上一条同类型」是**分面过滤+采样**问题,
不是语义相似问题。所以资产分两层:

- **数据层 = 结构化原子 schema**(§2):AI 能用的最小单位,字段可过滤、可采样、可审计。
- **索引层 = 分面索引(主)+ 语义向量(辅)**(§4):向量只干三件事——话题接梗、
  近义聚类去重、跨库查重。

几万条量级:SQLite 单文件 + 本地中文 embedding(bge 系)足够,**不引入向量数据库服务**。

## 1. 管线五步(每步产物落仓,可独立抽审)

```
raw jsonl(采集层,只增不改)
 → ①规范化:剥编号/OCR显错修正/本仓重算签名(采集层去重降格为粗筛,权威去重在此)
 → ②结构化抽取:LLM 批量把每条原文填进原子 schema(§2),宁空毋编
 → ③机制级聚类:按〈触发-动作-结算-拓扑〉embedding 聚类,近义合并为规范原子+变体表
 → ④打标归族:41族预归/野度1-10/S位安全旗/道具/场景/人数适配;与341表、15机制库跨库查重
 → ⑤建库:atoms.sqlite + embeddings.npy + 抽样器,三个消费方各走各的查询面
```

## 2. 原子 schema v0(AI 能用的最小单位)

```jsonc
{
  "atom_id": "xhs-00001",
  "name": "厕所门神",                  // 短名,可生成;原帖有名用原名
  "text_raw": "去厕所门口说…",          // OCR 原文,永不改动
  "source_ref": {"file": "raw-玩法-xhs-20260718.jsonl", "raw_record_index": 1},
  "atom_type": "条件点名|任务内容|完整玩法|问答题目|道具挑战|规则修饰",
  "trigger":   "何时/对谁触发(原文有才填)",
  "action":    "做什么",
  "judge":     "谁判/怎么判(可空=事实自明)",
  "settlement":"输赢/罚则落点(可空→内容原语)",
  "topology":  "全员|轮流|点名单人|两两|分队|null",
  "players_min": 2, "players_max": null,
  "props": ["酒","纸巾"],              // 空数组=无道具
  "scene": ["酒桌","KTV","车上","宿舍","露营","通用"],
  "wildness": 4,                       // 野度1-10,对齐口味滑块
  "safety_flags": ["饮酒","身体接触","异性互动","逼量嫌疑"],  // 对齐04 S位纪律
  "family_hint": null,                 // 41族预归,拿不准就null
  "confidence": "high|low"             // low 进人工复审池,不进主库
}
```

**铁则(吃 mechanism 模板事故的药):抽取只许「从原文里挑」,不许「替原文补」。**
judge/settlement 原文没写就留空——空值本身是信息(内容原语层的定义)。
每批抽样 30 条对原文抽审,发现补编即整批退回。

## 3. 三个消费方的查询面(最大化利用的具体形态)

| 消费方 | 查询形态 | 用哪层索引 |
|---|---|---|
| **狂野模式 `draw_atom`** | 分面过滤(type/人数/道具/场景/野度≤档/S位排除)→ 新颖度采样:排本局已用、同簇不连抽、全局使用频次衰减 | 分面为主;「接梗」子调用(他们在聊前任→前任类原子)走向量 top-k,再过分面闸 |
| **Mode A 生成佐料** | 分层采样:「给 20 条,横跨 ≥8 个 family/atom_type,野度按输入档」注入生成 prompt——把反同质化从「劝模型」变成「喂进去的分布本身就散」 | 分面+分层采样器 |
| **正典/机制库扩编** | ③聚类产物里「41族都挂不上的簇」= 新家族候选清单,走既有三裁/照准流程;高频簇 = 家族饱和度实测 | 向量聚类 |

## 4. 索引层实现(几万条的合理规模)

- `atoms.sqlite`:主表+分面索引(atom_type/scene/wildness/props/players)。
- `embeddings.npy` + id 映射:bge-small-zh 类本地模型对 `name+text_raw` 编码;
  聚类(③)、接梗检索、跨库查重共用这一份向量。
- 采样器(纯代码):新颖度加权 + 簇分散约束,~200 行 Python,无服务依赖。
- 全链跑在仓内脚本,产物文件化——与 check.py 同款审计姿势,版本戳+哈希照旧。

## 5. 施工顺序(与采集、引擎并行,不互卡)

1. **M-int-1**:①规范化 + ②结构化抽取跑通 4747 条(抽取用低价模型批跑,抽审 30/千)。
2. **M-int-2**:③聚类 + ④打标,产 `atoms.sqlite` v0 → **狂野模式 M3 的 draw_atom 换库点**。
3. **M-int-3**:Mode A 佐料采样器接入生成管线(对齐反同质化立案草案 C1)。
4. 采集线每来一批新 jsonl,增量走①–④,库只增不改;Codex 换平台不影响本线。

## 6. schema 实弹验证(2026-07-18)

用 `raw-玩法-xhs-20260718.jsonl` 前 60 条真实数据跑 50 条抽取(Haiku,与生产
路径同款成本),产物 `inputs/atoms/atoms-sample-v0.jsonl`。审计实测(全量核验,
非转述):

- **text_raw 50/50 与原文逐字相符**;variants 为唯一载荷的处置令被遵守。
- **宁空毋编守住**:settlement 非空 30 条全部对应原文含"喝/罚"的条目,补编 0;
  settlement 空 20 条即内容原语层,与两层论一致。
- atom_type:任务内容 25 / 道具挑战 24 / 条件点名 1(前 60 条系惩罚清单帖,分布合理);
  wildness 1–7 全用上;safety_flags 饮酒 34 / 身体接触 12 / 异性互动 8。
- **三处工艺缺陷,批量施工前修**:①confidence low = 0/50——低价模型不会自报
  不确定,须改**规则强制**(结构字段空 ≥3 项、或 text_raw 含 OCR 乱码特征即强制
  low 进复审池),不能靠模型自觉;②name 质量弱("厕所""冰块"式),生产批规范
  改"动词+宾语"式短名(如"冰块贴身""门口迎宾");③**props 0/50**——原文有
  "冰块""纸巾"却未结构化,修法=预编 20 词道具词表辅助识别,并分 `props_explicit`
  (原文明写)/`props_inferred`(从动作推断)两档,推断档不算原文事实。
- 抽取器另提两个字段候选(采信一驳回一):`trigger_type`(immediate/conditional/
  implicit,采信——70% trigger 空值里"立即执行"与"真缺失"应区分);
  「LLM 批量补全 settlement/trigger」**驳回**——补全即补编,违铁则,空值是信息。

**结论:schema v0 实弹通过,可按 §5 M-int-1 批量施工(附上述三条工艺修正)。**
