# 裁定纪要 · 按住说话(PTT)获批(2026-07-24,房主裁定)

## 裁定

**按住说话(Push-to-Talk)获批**:玩家按住「局长」键说一句,松手后由全模态模型
转成文字,以 `say(to=局长, via=voice)` 走既有 push_event 进事件流。仅此一路,仅此范围。

## 法理:为什么它不违反感知线收束令

感知线收束令(docs/records/裁定-感知线收束-20260720.md)禁掉的是**常驻/隐式采集**:
全程录音、笑声信号、能量窗口、分贝监听——采回来全是脏信号,且玩家不知道自己何时
被听。**那条禁令依然有效,一个字不改。**

PTT 属于另一族:**显式判定动作家族**(与拍照判定、录音判定、开局拍场同族)。
判据四条,全部满足才算这一族:

1. **显式动作**:玩家自己按下去,知道自己在说、在对谁说;
2. **按住才收**:不按不采一个字节,PTT 之外零采集;
3. **松手即停**:录音随手指抬起立即停止并上传,没有"顺带多听两秒";
4. **不留音频**:服务端转写完即弃——不落盘、不进 episode、不进任何缓存;
   事件流与 episode 里只有转写后的 say 文本,和打字发出的一模一样。

## 获批范围与边界(后来 agent 逐条对照,越界即违宪)

- **仅按住那几秒采集**:采集窗口 = 手指按住「局长」键的时长,别无其他窗口。
- **只有 to=局长 一路**:语音转写的去向永远是定向发言(享受既有遮蔽:别人的
  view 只见「有人跟主持说了句话」,不见内容)。**没有 to=桌上 的语音**——
  桌上说话是真实对话,发生在人和人的嘴与耳朵之间,系统永不掺和、永不转写。
- **音频不留存**:/api/stt 收到的 audio_b64 在转写返回后即弃。任何"顺手存一份
  方便调试/复盘"的念头都是违宪,复盘看 episode 里的转写文本即可。
- **转写口不可用(无 key)时**:501「语音通道未接入,打字仍可用」——PTT 是
  打字的语音化替身,不是新感官;打字那条路永远在。
- **客户端同责**:App 只在长按期间持有 Recording,松手/离场即停即弃;
  录音判定(judge.audio)的 recording 状态与 PTT 完全独立,互不复用。

## 防先例声明

本裁定**不是**监听的口子,**不可**被引作以下任何方向的先例:

- 常驻/后台/自动触发的录音或转写(不论时长多短、多"智能");
- 把 PTT 窗口悄悄拉长(松手后延录、按前预录、VAD 自动起录都算);
- 把语音转写扩到 to=桌上(桌上对话是产品的禁区,不是待开发区);
- 保存、缓存、回放任何原始音频。

凡引用本卷宗立项的功能,先逐条过上面"显式判定动作家族"四判据;
过不了四条的,归感知线收束令管辖——**不做,从立项里除名**。
