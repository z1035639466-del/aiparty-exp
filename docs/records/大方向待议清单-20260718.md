# 大方向待议清单(2026-07-18,Fable 盘点)

> 今日已裁完:双模边界/黑板架构/调用协议/原子库双轴/感知原语/评测转轨/采集续采。
> 下列四题为**未议且会反改设计**的大方向,按"越早裁越省返工"排序。

## ① 上桌形态与分发渠道(最先裁——它决定工程量的一半)

- **一机公放 vs 全员入网(多机私显)**:六机模拟台默认了多机;但多机才有
  私密显示(隐藏身份/密语/仅 show 本人),一机公放则这半个机制库直接失效。
  341 表大量 G2✓(仅show本人私密)押的都是多机。→ 需明裁:**多机是产品前提
  还是可降级选项?**降级模式(一机局)要不要在 draw_atom 加分面?
- **载体:App / 微信小程序 / Web(H5 扫码即入)**:派对场景"30 秒全员入局"
  是生死线——装 App 的摩擦可能直接杀死开局。Web/小程序扫码 vs App 的
  实时通道与多媒体能力,工程与分发的根本取舍。
- 附带裁定项:局长机(房主手机)与玩家机的角色分工;无网/弱网场景兜底
  (Mode A 件可离线,狂野必须在线)。

## ② 内容红线与合规(它决定野度滑块的上限刻在哪)

- 酒精内容:App Store / 微信平台对饮酒引导类内容的审核尺度不同,**分发渠道
  与野度上限互相锁定**——先裁渠道才能裁红线。
- 擦边内容:采集"照单全收"是语料纪律,**上桌是另一道闸**——需要一条
  "库里有 ≠ 能抽出"的发布过滤线(S 位+野度+渠道档三重闸,机制已具备,
  缺的是"每渠道的合规档位表")。
- 未成年人与饮酒责任:进场年龄声明?酒替代结算(罚饮→罚表演)一键切换?
  ——这个开关同时是市场扩量键(学生局/家庭局全靠它)。

## ③ 商业闭环与数据飞轮(壁垒到底是什么)

- 原子库可被竞争者重采,**真壁垒是 episode 飞轮**:玩得越多越知道什么好笑
  (哪条原子在哪种桌上被笑/被"过"),这个数据没人能抄。→ 裁定项:
  episode 埋点从 M1 第一行代码就带上(笑声/跳过/时长/复玩四信号),
  不是 M3 再补。
- 谁付费:局长订阅?按局?酒吧/桌游店 toB 授权?先不定价,但**计费单元**
  (局/桌/月)会影响账本与 session 的数据结构,M1 前最好有倾向。

## ④ 门槛验收单(D 件,总闸,悬置第 3 天)

- 正例反例各 3–5 个仍欠。现在所有照准件(R 口径/反同质化/架构案)都在
  "待房主照准"堆里,验收单是它们共同的对表基准。
- 建议降低交付成本:不用写文档——**语音说一段/举几个游戏名/贴几条觉得
  "对味"的小红书原帖**都行,Fable 负责翻译成可对账条目。
