# 狂野模式(活局长)· 架构立案草案(2026-07-18)

> 承接:`活局长-方向立项.md`(双模定案 §31:六原语即工具 API、账本写入仅经工具、
> state pad、≤3句+≤2工具/回合)与 `狂野模式-活局长prompt-v0.md`(裸奔版)。
> 本案回答两问:①最好的架构是什么;②为什么必须 agent 化(房主直觉的成文)。
> 性质:立案草案,待房主照准;不动 Mode A 任何冻结资产。

## 一、为什么必须是 agent——四条物理理由(房主"得建 agent"直觉的成文)

1. **账本铁律逼的**。"账本之外无事实"是全项目第一铁律,而 r2 实测散文判定的病灶
   就是"口头/意图/状态无通道"18/26。让模型用散文记分 = 把铁律交给它的自觉;
   让它**只能**经 state/score 工具写账 = 僭越在接口层物理不可能(立项原话)。
   工具化账本就是 agent 的核心形态:模型提议,工具执行,状态在模型外。
2. **状态漂移逼的**。长局(>30 分钟)模型必忘分、必前后矛盾(立项风险照实栏
   自认)。解法只有外置权威状态(state pad),模型每回合读摘要——外部记忆
   正是 agent 区别于长 prompt 的定义性构件。prompt v0 的"心里记个大概"只在
   30 分钟裸奔局成立,超时即崩。
3. **素材调用逼的**。已裁定(2026-07-18):原语靠采集不靠自编,模型分工是
   选择/组合/时机。4747 条(且持续增厚)的弹药库进不了 context,终态必是
   检索工具调用——"模型+检索工具"又是 agent。
4. **评测留痕逼的**。Mode B 没有 JSON 件可判级,评测转轨为 episode 留痕
   (立项已定:工具调用流水=天然日志,判级裁判转任复盘员)。散文聊天局
   无流水可审;只有工具化 agent 的调用序列天然就是可复盘的 episode。

四条合力 = 模型外面必须有一层运行时(感知→决策→工具→账本→循环)。
这层运行时就叫 agent。**故裁:狂野模式必须建成 agent,prompt v0 降格为
其人格层(system prompt 的语气段),不是替代品。**

## 二、荐案架构:单场内 agent + 四层,离线 agent 群供弹

### L0 会话内核(权威回合循环)
- 一个状态机持有**权威账本**(score/round/holder 起步)、玩家名册、野度档、已用原子表。
- 模型每回合输入 = 状态摘要 + 最近事件窗 + 佐料;输出 = ≤3 句话(流式播报)+ ≤2 工具调用。
- 延迟预算 1–3s(立项风险栏),**现场只此一个模型循环**——多 agent 会谋杀节奏,
  agent 化的富余全放离线侧。

### L1 工具面(七件)
- 六原语 show / ask / random / timer / state / fx + state pad(立项原封)。
- **新增第七件 `draw_atom(atom_type, 野度, exclude_used)`**:从弹药库抽原子
  (条件点名/任务内容/环节玩法),自动排已用、按野度滤档。
- **并行关键**:序列钉死"建库调用最后领工作证"——故 `draw_atom` **接口先定,
  库后换**:M1 期库=手拌佐料 10–20 条的静态数组(立项原案),整合线交付后
  换成检索库,**接口零改动**。这就是"边采原料边打引擎"的并行点,两线唯一
  的接缝就是这一个函数签名。

### L2 钳制层(代码层,不靠模型自觉)
- 铁律段 + 野度滑块 + 判定三层 + 房主修正权(立项沿用清单)拦在**工具调用与
  账本之间**:score 增减越界拒写、未经 ask/consensus 确认的判定拒写、"过"字
  即时短路当前原子。prompt 层劝,钳制层拦——两层不互替。

### L3 复盘层(离线)
- episode.jsonl 全量落盘(每回合:状态摘要/模型输出/工具调用/账本差分)。
- 判级裁判 v0.3 转任复盘员(立项已定)+ 真人局四问;笑声账本口径可直接复用。

### 离线 agent 群(已在运转,非新建)
采集(Codex)/ 整合聚类(embedding,待上岗)/ 探针(族频次·骨架·查重)/ 复盘。
项目整体形态:**现场一个薄 agent,周围一圈离线 agent 供弹与审计,房主唯一裁定人。**

## 三、里程碑切法(每档独立可玩,不齐不发)

| 档 | 内容 | 依赖 |
|---|---|---|
| M0 | prompt v0 裸奔局(已有,即贴即用) | 无 |
| M1 | L0+L1:六原语+state pad+手拌佐料数组版 draw_atom | 无(可即开工) |
| M2 | L2 钳制层 + L3 episode 落盘 | M1 |
| M3 | draw_atom 换检索库 | 整合线交付(atom_type 分层+聚类) |

试玩协议照立项:同一张三人桌,Mode A 四绿件 vs 活局长一晚,四个二值+一句话,桌子说了算。

## 四、双模边界裁定(2026-07-18 追加:狂野与普通要不要分开设计)

**裁:分开设计"决策源与验收",共用"执行底座"——两个大脑,一副身体。**

| | Mode A 普通(预编译) | Mode B 狂野(活局长) |
|---|---|---|
| **决策源(分开)** | 编译产物:flow/rules/settlement 确定性执行 | 模型回合循环(L0),现场决策 |
| **验收(分开)** | check 硬闸+判级裁判+golden(冻结资产照旧) | episode 复盘+真人局四问(立项已定转轨) |
| **失败观(分开)** | 结构违规=拒件,编译期零容忍 | 翻车观:自嘲翻篇、房主一锤(prompt v0) |
| **执行底座(共用,只建一次)** | 账本状态机 · 六原语工具面 · state pad · 钳制层(野度/S位/修正权/"过"短路) · draw_atom+原子库 · episode 留痕格式 · 笑声账本口径 | 同左 |

三条裁定理由:
1. **语义漂移**:账本与六原语若两套实现,同一规则在两模下迟早不同判——
   "账本之外无事实"会裂成两本账,这是底座必须唯一的硬理由。
2. **评测可比性**:立项试玩协议就是**同一张桌 Mode A 四绿件 vs 活局长**对比;
   底座不同,对比测的就是两套引擎的差而不是两种模式的差,协议失效。
3. **工程减半**:钳制层、留痕、原子库接入各只写一次,两模白得。

实施口径:
- 共享底座以 **Mode B M1 为首建**(新起,干净);既有 v0 引擎**不强迁**,
  引擎线动"传递链/转移"能力缺口时择机并轨到同一工具面。
- 边界纪律双向:Mode A 编译期资产(check/params 硬闸/判级)按立项"封存待命"
  原则**严禁泄入** Mode B 运行时(活局不写 JSON、不判级);反向,Mode B 的
  "现场改规则"自由度也严禁泄回 Mode A 的确定性执行。
- 远景留一笔不立项:混合局(Mode A 件跑崩时活局长接管救场)——底座同一才有
  这个可能性,分开设计就永远做不了。

### 四·补:现场反馈的四个适应杠杆(2026-07-18 房主问「预编译是不是没法随现场调整」)

**Mode A 局内不读场是特性不是缺陷**——可判级、可复现、质量有闸(裁判为第 100 桌
把关)全系于此。但"随现场调整"分四个时间尺度,预编译只放弃了其中一个:

| 杠杆 | 时间尺度 | 谁做 | 状态 |
|---|---|---|---|
| ①预置分支 | 局内·预料中的变化 | Mode A 件自身:tiebreak/超时/「过」跳过/人数适配/野度参数 | 已有,skill 正典范畴 |
| ②读场选件(**DJ 层**) | **局间** | 轻量模型循环:读上一局的笑声/节奏/弃局信号 → 从件库选下一件、调野度档、换快慢型 | **新立,双模底座的免费红利** |
| ③现场改规则/接梗 | 局内·预料外 | 活局长(Mode B)专属;Mode A 永不做,做了即丧失判级根基 | 立项已定 |
| ④数据回流 | 跨局/跨晚 | 笑声账本+episode 复盘 → 选件权重与下一版生成佐料 | 评测转轨既定路线 |

**DJ 层(②)是本问催生的新构件**:一晚不止一局——局内 Mode A 确定性执行原封不动,
局与局**之间**由模型读场选件,正是 prompt v0「像 DJ 调歌一样调游戏」那句话的
工程化落点。它跑在共享底座同一工具面上(读 state pad 摘要,调 draw 件而非 draw 原子),
比整局活局长便宜一个量级,且不碰 Mode A 任何冻结资产——**可作混合形态的第一块
踏脚石,先于完整 Mode B 上桌**。归 M2 档后评估立项。

## 五、为什么不只用狂野模式(2026-07-18 房主之问「感觉全面优于普通模式」)

**裁:不成立——狂野是"上限型",普通是"下限型",各占一头,故双模。**

| 维度 | 普通(预编译) | 狂野(活局长) |
|---|---|---|
| 质量下限 | check+裁判+重试离线淘洗,精品件下限有闸 | 模型当晚发挥,无闸。r2 实测:模型自由发挥语义 pass@1 四模型全 0;离线拒件成本≈0,现场翻车成本=冷场 |
| 复杂结构 | 隐藏身份/多阶段结算可承载(账本+编译期为此而生) | 30 分钟即漂移(立项自认),只适合轻结构原子玩法 |
| 成本/延迟 | 编译后本地执行,零 API 零延迟 | $0.3–0.6/局,每回合 1–3s,节奏被 API 绑架 |
| 资产沉淀 | 件=产品,打磨一次上万桌,可复现可迭代 | 每局蒸发只留 episode,是表演不是产品 |
| 内容预审 | 编译期全过审(野度/S位/羞辱闸) | 钳制层拦得住账本拦不住嘴,现挂话术无法预审 |
| 评测 | 离线批测,便宜 | 只能上桌评,贵且慢 |
| 适应性 | 局内仅预置分支 | 读场/接梗/现改规则,全场脱稿 |

正确用法即 §四 的三层:**精品件保下限,DJ 层调现场,活局长保上限**——砍掉任何一头,
要么失去"永远不难看",要么失去"偶尔很惊艳"。

### 五·附:轮次场数考(同日房主之问)

查实:用户 input(`inputs/input_A.json`)只有 venue/players/口味/素材四项,**无轮数字段**;
轮数是**生成模型设计时自定**(如 humanpilot_01:3 人 3 轮一人一轮)并写死进 flow。
skill 强制的是"必须有明确终局"(设计定律:到顶强制摊牌),**不强制具体轮数**。
故"固定轮次"是单件属性、非全局规则。想要不固定,三个口子由轻到重:
①件内预置分支(加时轮/提前终局条件,正典已有形态);②DJ 层局间"再来一局";
③狂野模式天然无固定轮次(prompt v0 的"30 分钟终局仪式"是节奏建议非硬轮数)。
若要让用户 input 直接控轮数/时长,属 skill 输入 schema 变更,排队走照准,本案不动。

## 六、风险照实(沿立项,不设门只记账)

单局 $0.3–0.6 · 回合延迟 1–3s · 长局漂移(state pad 缓解)· 评测重启成本;
新增:draw_atom 抽中臭原子的现场成本(钳制层"过"字短路 + exclude_used 缓解,
臭原子回流打标供整合线降权)。
