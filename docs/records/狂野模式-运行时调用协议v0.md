# 狂野模式 · 运行时调用协议 v0(2026-07-18)

> 回答房主之问:API 传什么/模型回什么/谁来执行/是不是每条反馈都走 API/
> 模型"调整"的到底是什么东西。并纳同日房主四裁定(§六)。
> 承接:`狂野模式-架构立案草案.md` L0–L3。性质:协议草案,M1 施工依据。

## 一、总图(一句话)

模型不直接碰任何东西。**上行** = 当前局面;**下行** = ≤3 句主持词 + ≤2 个
结构化工具调用**意图**;**执行者永远是我们的运行时**:校验(钳制层)→ 执行
(写账本 / 下发玩家端指令)→ 结果进下一回合上行。如此循环。

## 二、上行:我们通过 API 传给模型的三段

1. **system(每局固定,吃 prompt 缓存)**:人格层(prompt v0 语气段)+ 铁律段 +
   野度钳制 + 本桌名册与关系梗。
2. **tools 声明**:七件工具的 JSON schema——show / ask / random / timer /
   state / fx / draw_atom。模型只被告知"你能提议这些动作"。
3. **本回合消息**:状态摘要(score / round / 焦点人 / 已用原子 / **剩余时长**)
   + 自上一回合以来的**事件窗**(聚合,见 §四)+ 按需的 draw_atom 返回佐料。

## 三、下行:模型传回的两样

- **text**:≤3 句主持词,流式播报到各玩家端。
- **tool_use 块**:≤2 个,带 JSON 参数的调用意图,如
  `{"name":"state.add_score","input":{"player":"小静","delta":1,"reason":"冰块挑战过"}}`。

模型回传的是**意图,不是执行**。运行时是唯一执行者:钳制层校验(分数越界拒写、
S 位/野度过滤、"过"字短路)→ 写账本 → 下发 UI 指令 → 记 episode 一行。

**「API 调整的东西是不是 JSON 文件?」——不是。狂野模式没有 JSON 件。**
被调整的只有两样:①**账本状态**(唯一入口 = state 工具);②**玩家端界面**
(show/ask/timer 指令下发到六台手机)。JSON 只以"工具调用参数"的形态存在,
它是电报,不是文件。

## 四、什么走 API、什么不走(成本与延迟的关键)

**不是每条反馈一次 API。**事件分两类:

| 类别 | 例 | 处理 |
|---|---|---|
| **规则内事件(本地,零 API)** | 投票计票、tap 确认、计时到点执行既定结算、"过"字跳过、比分展示刷新 | 运行时状态机直接办,顺手记 episode |
| **决策点事件(聚合 → 一次模型回合)** | 该主持人开口(开场/转场/宣布结果/接梗)、需要判断(争议、意外输入)、该选下一个活 | 两回合之间事件入队,**一次性**作为事件窗上行 |

粗算:30 分钟局 ≈ 40–60 个决策回合——立项 $0.3–0.6/局即由此来。
"玩家每次点击都过 API"的设计是错的:延迟和成本都放大十倍,且模型被琐事淹没。

## 五、一个完整回合(实弹格式示例)

上行(第 N 回合消息,附在缓存的 system+tools 之后):

```jsonc
{ "state_digest": {"round": 4, "scores": {"疯子明":2,"小静":1,"大鹏":3},
    "focus": "小静", "atoms_used": ["xhs-00002"], "time_left_min": 18, "野度档": 6},
  "events": [
    {"t":"+0s","type":"challenge_done","player":"小静","atom":"xhs-00002 冰块贴身"},
    {"t":"+8s","type":"vote_result","pass":true,"tally":"2:1"} ],   // 计票已在本地完成
  "draw_result": null }
```

下行(模型响应):

```jsonc
{ "text": "小静这个表情我建议申报吉尼斯。二比一,过了!下一位受害者让骰子决定。",
  "tool_use": [
    {"name":"state.add_score","input":{"player":"小静","delta":1,"reason":"冰块挑战过"}},
    {"name":"random.pick","input":{"from":"players","exclude":["小静"]}} ] }
```

运行时随后:钳制校验 → 账本 小静+1 → 掷选下一人 → 结果进第 N+1 回合事件窗 →
episode 追加一行(digest/text/tool_use/账本差分)。

## 六、房主四裁定纳入(2026-07-18,原架构案两处表述随之修订)

1. **漂移辩证**:随机性是线下娱乐的乐趣来源——内容层(random/draw_atom/现挂/
   接梗)**不设防,越野越好**;要兜的只是**账本层**漂移(忘分、前后矛盾),那是
   公平感杀手,由 state pad 兜底。翻车观照旧:忘梗可以自嘲炒成菜,分数不能错。
   架构案 §五 对照表"漂移"行语义**收窄为"账本漂移"**。
2. **「狂野每局蒸发」表述作废**:episode 流水是迭代燃料,回流四路——提示词迭代
   (哪类回合冷场)/ 检索优化(哪类原子老被"过")/ 原子降权与**自产收割**
   (活局长现挂出的好玩法反向入原子库,标 `source=live`)/ 软件模式本身迭代。
   "把把惊艳"就是靠这个循环逼近的——这是最终目标的工程路径。
3. **开局配置补一项:想玩多久(时长预算)**。人数已有(名册),时长预算入状态
   摘要(`time_left_min`),模型据此控节奏、收终局;prompt v0 开局 30 秒问清单
   同步补"玩多久"。Mode A skill 输入 schema 不动。
4. **评测转轨:六机模拟台优先,真人局降为最后验证**。M2 交付「六台模拟手机
   操控面」:浏览器一页 = 6 个玩家端 pane + 房主视角 + 账本/episode 实时窗,
   房主一人操六机找实际问题。语序:模拟台 → 迭代循环(提示词/检索/道具)→
   真人局只做最终验证。**基础没搭好之前不上人,与门槛裁定同源。**
